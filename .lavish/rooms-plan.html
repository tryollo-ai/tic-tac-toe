Game Rooms for Tic-Tac-Toe

Implementation plan · server-backed rooms with live spectating

Today the app opens straight into one game, with all state in local React hooks and no backend. This plan adds a lobby of shared rooms: you land on a list of rooms, drop into any one to play or watch, and see games update in near‑real‑time. The server becomes the single source of truth; the existing game logic and UI components are reused, not rewritten.

Server owns state
A Map<id, Room> stashed on globalThis survives dev hot‑reload. All mutation & validation lives in roomStore.ts; handlers stay thin.
AI runs server-side
In vs‑AI rooms, seat O is the AI. After your move the server applies getBestMove, so spectators see it too. The old client AI timer is removed.
Polling, not sockets
A small usePolling hook (no new deps). Your own move is optimistic; polling pauses during the in‑flight write so a stale GET can't clobber it.

The Room object

// lib/roomTypes.ts
interface Room {
  id: string
  name: string
  board: Board        // Cell[9]
  xIsNext: boolean
  scores: { X, O, draws }
  status: "waiting" | "in-progress" | "finished"
  seats: { X: id|null, O: id|"__AI__"|null }
  seatSeen: { X, O }  // heartbeat ts
  mode: "two-player" | "ai"
  createdAt, lastActivity
}
The move rule
On POST /move the server checks, in order:
1 · room exists → else 404
2 · not over → else 409 game-over
3 · seats[turn] === playerId → else 403
4 · cell empty → else 409 cell-taken
5 · apply, flip turn, score if finished
6 · if vs‑AI & O's turn → apply getBestMove

07

Edge cases handled

Both claim X at onceHandlers run sequentially (single‑threaded, no await mid‑mutation). First wins; second gets 409 seat-taken.
Move out of turn / wrong seatBoard is disabled client‑side; server is the guard and returns 403, then UI resyncs.
Room not foundAfter restart or bad link → 404; room page shows a recovery link, lobby poll drops it.
Leave seat (intentional)"Leave seat" button → DELETE /seat. Seat → null, board/turn preserved; a new visitor can claim and continue. Board disabled until the on‑turn seat is filled.
Close tab / lose connectionPresence heartbeat. Each room poll bumps seatSeen[playerId]. The server lazily auto‑releases any seat unseen for > 30s, so an abandoned seat frees itself. Best‑effort instant release on tab close via navigator.sendBeacon → DELETE /seat.
Reopen after leavingSame playerId in localStorage: if the seat wasn't reclaimed yet, you resume it; if someone else took it, you become a spectator.
Spectator clicks boardNo‑op client‑side, rejected server‑side.
Stale poll vs your movePolling pauses during your in‑flight write; the POST response is authoritative and replaces local state.

09

How we'll verify it

End-to-end with two browser windows
  • npm run dev → open /: empty lobby → create a room → redirected to /room/[id].
  • Browser A "Play as X", Browser B "Play as O". A moves only on X's turn, B only on O's; each sees the other within ~1.5s.
  • A third tab spectates — sees moves live, board disabled.
  • Create a vs‑AI room, claim X, move → AI (O) responds within a poll cycle; a spectator sees both.
  • Confirm 409 seat-taken, 403 not-your-turn, finished‑game, and 404 behaviors.
  • Close the X player's tab without leaving → within ~30s the seat frees and another browser can claim X and continue the same game.
  • "New Game" resets the board but keeps scores; lobby previews update live.
  • npm run lint and npm run build pass under TS strict.
