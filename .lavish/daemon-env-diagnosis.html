Why none of my agent-step wiring helped

What my env: reaches

UNREACHABLE

The no-mistakes daemon

A separate systemd --user service. The CLI just talks to it over a socket. It spawns the review claude, git push, and gh.

env: HOME + PATH only
1 · Not a child of the agent

Step-level env: propagates to the agent's own subprocesses - not to a pre-existing daemon it merely connects to. So the token I set never lands where the review step runs.

2 · The token is scrubbed

Claude Code deliberately strips CLAUDE_CODE_OAUTH_TOKEN from the agent's Bash subprocesses (…ENV_SCRUB). The agent literally can't read it to hand it on - which is why it spelunks /proc and ALL_INPUTS to recover it every run.

The fix moves out of the agent, into a workflow step

Prime the daemon before the agent runs, in a normal step whose env is not scrubbed.

1
Global git credentials

git config --global url."https://x-access-token:$APP_TOKEN@github.com/".insteadOf … writes to ~/.gitconfig, which the daemon's git (same HOME) reads.

2
Inject env via the systemd user manager

systemctl --user import-environment CLAUDE_CODE_OAUTH_TOKEN GH_TOKEN GITHUB_TOKEN PATH - so the daemon inherits them even across the unit regeneration that defeated the agent's hand-edits.

3
Restart the daemon once, deterministically

It picks all of it up. The agent then just calls no-mistakes axi run and it works - no mid-pipeline surgery.

Second wall - now cleared: the App has the permissions to push.

The App (ollo-ai-board-sync[bot]) now carries read/write on code, issues, org projects, pull requests, and workflows (+ metadata read) - the full set. The 403 that killed last run's push is resolved. With checkout already wired to mint and use this App's token, the only thing left between the loop and a green run is the daemon.
